# Архітектура AiM Skill

Управлінський облік для ТОВ «ЕЙМ СКІЛ» (ЄДРПОУ 45505924). Цей документ описує, **що є в системі і як воно пов'язане**.

> Мета системи: **проста** в користуванні, дає **об'єктивні взаємопов'язані** дані для управлінських рішень. Не статутний облік, не план рахунків, не зарплата.

---

## 1. Технологічний стек

| Шар | Технологія |
|---|---|
| Фронтенд | React 18 + Vite 5, роутинг через `react-router` |
| Бекенд | Supabase — Postgres (БД), Auth (вхід), Storage (файли в приватному бакеті `documents`) |
| AI / OCR | Anthropic Claude (Sonnet 4.6) — розпізнавання документів і парсинг тексту |
| Графіки / експорт | Recharts (діаграми), SheetJS/`xlsx` (Excel) |
| Стилі | Без CSS-фреймворку — стилі як JS-шаблони в `src/lib/styles.js`, іконки Tabler з CDN |

Бізнес-логіка винесена в `src/lib/`:
- `ai.js` — OCR документів та парсинг тексту (Claude)
- `autoClassify.js` — правила авто-класифікації банк-операцій
- `stockService.js` — склад, рухи, FIFO-собівартість
- `articles.js` — статті доходів/витрат + ієрархія P&L (`pl_level`)
- `companyConfig.js` — реквізити компанії
- `supabase.js` — клієнт Supabase
- `fmt.js` — форматування чисел

Змінні середовища: `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`, `VITE_ANTHROPIC_KEY`.

## 2. Модулі (UI)

### Введення даних
- **Документи** (`DocumentUpload`) — завантаження скана → Claude витягує поля (контрагент, ЄДРПОУ, сума, ПДВ, позиції) → створює запис, прив'язує до банк-операції.
- **Банк / Звірка** (`Bank`) — імпорт виписки (Monobank / ПУМБ / PDF через Claude) → авто-класифікація → масове створення операцій.
- **Каса** (`Cash`) — окремий журнал готівки: прихід / видаток / підзвіт / перекази банк↔каса.

### Облік / звірка
- **Реєстр** (`Registry`) — головний журнал усіх банк-операцій: фільтри, масове редагування, пошук дублів, відновлення позицій.
- **Валідація** (`Validation`) — розбиття на ПДВ, перевірка сум, прапорець `is_validated` (без нього дані **не йдуть** у звіти).

### Звітність (`Analytics`)
Вкладки:
- **Огляд** — KPI, графіки, підсумок дебіторки/кредиторки.
- **P&L** — звіт про прибутки по статтях/місяцях. Перемикач **Факт / План / Порівняння**.
- **Бюджет і прогноз** — зверху редагований план P&L; знизу грошовий прогноз, що оновлюється з плану.
- **Заборгованість** — старіння (aging) неоплачених документів.

### Операційні модулі
- **Замовлення** (`Orders`) — Kanban-пайплайн замовлень + генерація документів (`DocGenModal`).
- **Контрагенти** (`Contractors`) — довідник, контакти, інтеграція Vkursi, AI-парсинг реквізитів із тексту.
- **Склад** (`Inventory`) — товари, рухи, FIFO-собівартість, об'єднання дублів.
- **Збірки** (`Assembly`) — складання готового продукту з компонентів (списання компонентів, оприбуткування результату).

### Адмін
- **Налаштування** — Реквізити · Користувачі (ролі) · Статті (P&L-класифікація) · Інтеграції.

## 3. База даних і зв'язки

**Центр системи — `bank_transactions`** (реальні гроші).

```
ЦЕНТР:  bank_transactions  ← головний реєстр грошей
            │
   ┌────────┼──────────────┬──────────────┐
documents  transaction_items   contractors   articles
(файли)    (позиції + ПДВ)     (довідник)    (статті P&L)
            │
       stock_movements ── products ── product_aliases
            │                  │
       assemblies /        product_stock (view)
       assembly_items
```

| Таблиця | Що зберігає | Хто пише | Хто читає |
|---|---|---|---|
| `bank_transactions` | усі банк-операції (сума, напрям, стаття, ПДВ, `is_validated`, `is_ignored`) | Банк, Документи | Реєстр, Валідація, **P&L**, Огляд |
| `transaction_items` | позиції документа + ставка ПДВ | Документи, Валідація | Валідація, Склад, P&L (нетто) |
| `documents` | метадані файлів у Storage | Документи, Каса | Реєстр, Валідація |
| `cash_transactions` | операції готівки | Каса | Каса, Огляд (баланс) |
| `articles` | статті доходів/витрат, `type`, `pl_level`, `sort_order` | Налаштування | усі звіти, класифікація |
| `contractors` | контрагенти (ЄДРПОУ, реквізити, дефолти) | Контрагенти, Банк, Документи | авто-класифікація, звірка |
| `contractor_contacts` | контактні особи | Контрагенти | Контрагенти |
| `products` | товари | Склад, Збірки, Документи | Склад, Збірки |
| `product_aliases` | синоніми назв товарів (для зіставлення) | Склад (авто) | зіставлення позицій |
| `stock_movements` | рухи складу (IN/OUT), FIFO | Склад, Збірки, Документи | Склад, Збірки |
| `product_stock` (view) | поточні залишки (рахується з рухів) | — | Склад, Валідація |
| `assemblies` / `assembly_items` | збірки та їх склад | Збірки | Збірки |
| `generated_docs` | документи, **згенеровані в системі** (накладні, акти, замовлення) | Замовлення, генерація | Заборгованість, прогноз, Замовлення |
| `plans` | бюджет (план P&L): `year_month`, `direction`, `article`, `amount`, шаблони | Бюджет | Бюджет, P&L Порівняння, прогноз |
| `projects` | проєкти | Налаштування / Контрагенти | прив'язка операцій |
| `profiles` | користувачі + ролі | Налаштування | контроль доступу (UI + RLS) |

## 4. Потоки даних

### 🟢 Об'єктивне ядро (працює надійно)
```
Скан → Документи (AI) ─┐
                       ├─→ bank_transactions ─→ Валідація ─→ P&L / Звітність
Виписка → Банк ────────┘    (+ позиції/ПДВ)     (is_validated)     │
                                                              Каса (окремо)
```
Заносиш факт → валідуєш → бачиш реальний прибуток. Усе пов'язано й об'єктивно.

### 🟡 Планування (надбудова над фактом)
```
Бюджет (ти вводиш план) → plans → P&L «Порівняння» (план vs факт)
                                 → Грошовий прогноз (план + борги + замовлення)
```

### 🟡 Операційні острови (працюють, але НЕ впливають на P&L)
- **Склад / Збірки** — рахують FIFO-собівартість, але P&L їх не читає (COGS у P&L поки не потрапляє).
- **Замовлення / Контрагенти / Проєкти** — операційний облік, фінрезультату не змінюють.

## 5. Відомі обмеження / борг

1. **Дебіторка/кредиторка неповна.** Заборгованість і прогноз читають **тільки** `generated_docs` (документи, згенеровані в системі). Борги з документів-**сканів** (які йдуть у `bank_transactions`/`documents`) туди не потрапляють. Щоб зробити правдиво — рахувати борги і з неоплачених сканованих документів.
2. **Склад → P&L не зв'язаний.** Собівартість зі `stock_movements` не використовується для COGS у P&L.
3. **Замовлення → Склад не зв'язані.** Замовлення не резервують і не рухають складські залишки.
4. `Reports.jsx` містить блок таблиці руху коштів під `initialTab==='cf'` — наразі недосяжний (мертвий, але нешкідливий).

## 6. Ролі та доступ
Чотири ролі в `profiles`: **admin / accountant / manager / viewer**. Контроль на двох рівнях — у компонентах (`user.role`) і в RLS-політиках Supabase. «Налаштування» доступні лише admin.

## 7. Зовнішні інтеграції
- **Claude API** — OCR документів (на проді через проксі `/api/ai`). Реквізити компанії зашиті в системному промпті `ai.js`.
- **Vkursi.pro** — дані контрагентів за ЄДРПОУ (через проксі `/api/vkursi`). Працює.
- **Monobank** — не реалізовано (і не підходить для ТОВ, лише для ФОП).

## 8. Деплой
Vercel з авто-деплоєм із гілки `main` у GitHub. Supabase — бекенд (БД, авторизація, файли в приватному бакеті `documents`).

*Підсумок одним реченням:* центр — `bank_transactions` (реальні гроші); навколо — введення (Документи/Банк/Каса), очистка (Реєстр/Валідація) і звіти (P&L/Бюджет); Склад/Замовлення/Контрагенти — окремий операційний контур, поки слабо зчеплений із фінансовими цифрами.
