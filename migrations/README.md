# Міграції — REWRITE v2

Стратегія: **additive еволюція на місці** того самого Supabase-проєкту, поки стара апка ще в проді на `main`. Нові колонки/таблиці стара апка ігнорує → продакшн не ламається. Переміщення рядків — окремо, при cutover.

DDL через PostgREST неможливе → SQL запускаєш вручну: **Supabase Dashboard → SQL Editor → Run**.

## Порядок

| Файл | Коли | Що робить | Безпечно для проду? |
|---|---|---|---|
| `001_phase1_schema.sql` | **Зараз** | 8 нових таблиць, ADD COLUMN до наявних, рахунки, індекси, в'юхи (`product_stock`, `contractor_balances`), RLS на нових таблицях, безпечні backfill (account_id, article_id) | ✅ так (тільки additive) |
| `002_cutover.sql` | При злитті `rewrite`→`main` | каса→bank_transactions, backfill сум документів, (опц.) projects→orders | ⚠️ ні — ризик подвійного обліку в старій апці |

## Як застосувати 001 (зараз)
1. Відкрий `001_phase1_schema.sql`, скопіюй увесь вміст.
2. Supabase Dashboard → SQL Editor → встав → **Run**. Має завершитись `commit` без помилок. Ідемпотентний — повторний запуск безпечний.
3. Локально: `node migrations/validate.mjs` — перевірка цілісності (дані збережені, таблиці є, backfill ок, в'юхи працюють).

## Примітки
- `bank_name` був заповнений лише у 66/476 транзакцій → решта (null) віднесена до рахунку **ПУМБ** (основний). Mono-транзакцій у наявних даних немає. Якщо частина null насправді Mono — скоригуй `account_id` вручну.
- `direction` лишається у P&L-семантиці (`Доходи/Витрати/ПФД/Інше`) — стара апка на неї покладається. Нова апка похідно обчислює in/out.
- Реальні **борги** запрацюють після заповнення `documents.amount` (backfill у 002 + OCR у Фазі 5 + Aging у Фазі 7). Зараз `contractor_balances.documents_total≈0` — це очікувано.
- `projects` (22) лишається недоторканим; рішення конвертувати в `orders` — у Фазі 4.
