## 1. Що це за система

**AiM Skill** — україномовна система **управлінського обліку** для **ТОВ «ЕЙМ СКІЛ»** (ЄДРПОУ 45505924). Переписана з нуля за ТЗ v2 (червень 2026).

**Три напрямки бізнесу:**
1. **Торгівля** комп'ютерною технікою — закупівля у дистрибюторів, перепродаж клієнтам.
2. **Послуги** — разові: рахунок → оплата → виконання → акт.
3. **Агентська винагорода** — посередництво між клієнтом і партнером за комісію.

**Головний принцип:** центр системи — `bank_transactions` (реальні гроші). Усе прив'язується до них. **Борги рахуються автоматично** (сума документів − сума прив'язаних транзакцій), ніколи не вводяться вручну.

**Post-launch покращення (зроблено):**
- Рахунки показують **реальний залишок** (усі неігноровані транзакції + початковий залишок), а не лише рух. Картка: Залишок + Надходження + Витрати.
- **Авто-матч контрагента** по ЄДРПОУ/назві при імпорті, авто-класифікації та відкритті непідтверджених.
- **Кнопка «Розпізнати»** для існуючих документів (OCR зі Storage → метадані).
- **Текст не обрізається** ніде (перенос; вузькі місця — ellipsis+title).
- **Авто-повтор OCR/виписок** на перевантаженні Claude API (529/429).
- **Редагування назви файлу** у формі OCR/розпізнавання (оновлює `documents.file_name`).
- **Номер документа** (`documents.doc_number`) — поле у формі OCR (префіл з OCR) + колонка «№» у реєстрі.
- **Фікс:** додано UPDATE-RLS політику на `documents` (її не було → «Розпізнати» тихо не зберігалось). Save кидає помилку, якщо UPDATE зачепив 0 рядків.
- **Редагування + видалення** документа в екрані перегляду (Видалити — для ролей admin/accountant; перевіряє, що рядок справді видалено).
- **Ручна прив'язка документа до руху складу:** кнопка 🔗 біля рухів без документа (Рухи + картка товару) → DocPickerModal з **рекомендацією за датою** (документи з датою близькою до дати руху, бейдж «рекомендовано · ±Nдн», сортування за близькістю) + пошук за назвою/номером.
- **Backfill рухів складу:** 190 рухів source='document' без document_id прив'язано до документів через bank_transaction (транзакція з 1 документом). Для решти 34 показується читабельна мітка джерела (з документа/вручну/збірка) замість сирого 'document'.
- **Склад → документ-джерело:** у вкладці «Рухи» і в картці товару рух показує прив'язаний документ (мітка типу) і кліком відкриває DocModal (138 рухів мають document_id).
- **Часткове покриття tx↔doc:** одна накладна на кілька платежів — прив'язка враховує залишок документа з усіх транзакцій (min(непокрито tx, залишок doc)); у вікні показано «залишок».
- **Дедуплікація документів:** видалено 20 повних дублів (контрагент+тип+номер+сума; 159→139), зв'язки переносяться на копію що лишається. Лишилось 4 групи з однаковим номером але різною сумою — на ручний огляд (ООН/ПРООН).
- **Сортування таблиць** (`components/Sort.jsx`: useSort + SortTh) — клік по заголовку. Застосовано: Документи, Контрагенти, Замовлення, Склад (Залишки+Рухи), Банк/транзакції. **Пошук у транзакціях** за контрагентом/ЄДРПОУ/описом (server-side ilike).
- **Прив'язаний документ у транзакціях:** колонка «Документ» у таблиці й секція в TxModal показують прив'язані документи (або «не прив'язано»); клік відкриває DocModal.
- **Банк: транзакції у табличному вигляді** (як Документи) — колонки Дата/Контрагент/Сума/Напрям/Стаття/статус, рядок клікабельний → модалка `TxModal` (класифікація, підтвердження, ігнор, прив'язка документів). Замість inline-рядків.
- **Сума документа = з ПДВ (gross).** `documents.amount` тепер повна сума (totalAmount), `vat_amount` — у т.ч. ПДВ. Раніше було net → борги занижені, матчинг по сумі не спрацьовував для ПДВ-документів. Backfill 143 документів. Авто-лінк зріс 14→85.
- **Авто-прив'язка документ↔транзакція** (`lib/docMatch.js`): матч за контрагентом + сумою (±0.5%) + датою документа (±15 днів) — усі три разом. У вікні прив'язки кандидати ранжовані з бейджами «сума ✓/дата ±Nдн»; кнопка «Авто-прив'язка» масово лінкує єдині впевнені збіги (рахунки на оплату — нижчий пріоритет, лінк радше на накладну/акт).
- **Захист від подвоєння боргу:** рахунок на оплату / замовлення / договір НЕ створюють борг (лише накладні та акти). Рахунок+накладна на одну угоду = один борг. `countsAsDebt()` у debts.js/pl.js.
- **Дата документа** (`documents.doc_date`, міграція 005) — окреме поле (раніше показувався created_at, дата з форми не зберігалась). Реєстр/вкладки показують doc_date.
- **Спільна модалка документа** (`components/DocModal.jsx`) — переюзана в Документах, картці контрагента і картці замовлення. У вкладках «Документи» тип показується мітками (не сирим ключем), клік по рядку відкриває перегляд з прев'ю/редагуванням/видаленням.
- **Прев'ю документа** у формі OCR/розпізнавання (двоколонкова: файл зліва, поля справа). Клік по рядку реєстру → той самий екран у режимі перегляду (прев'ю + збережені поля, без авто-OCR; кнопки «Розпізнати»/«Зберегти»).
- **Інтеграція корпоративної пошти (Hostinger IMAP/SMTP)** — серверні ендпоінти в `api/` (патерн `api/ai.js`), залежності `imapflow`/`nodemailer`/`mailparser` (лише серверні). **Концепція (фінальна):** окремий розділ **«Пошта»** — вхідні листи НЕ створюють документи автоматично; заявка створюється з листа окремою дією.
  - **Розділ «Пошта» (`pages/Mail.jsx`, маршрут `/mail`, 9-й розділ навігації):** список усіх листів (вхідні/вихідні, незалежно від вкладень), перегляд листа (тіло в sandbox-iframe, вкладення через signed URL), кнопка **«Створити заявку»**.
  - **Синхронізація (`api/mail-sync.js`):** INBOX (`in`) + Sent (`out`, папка шукається за specialUse/назвою) за **останні 10 днів**, вкладення → Storage (`email/<msgid>/`), рядки `emails`. **Документів не створює, листи прочитаними НЕ помічає.** Дедуплікація за Message-ID. Батч `MAX_NEW=40`/запуск (truncated→натиснути ще). Тригери: кнопка «Синхронізувати», Cron (раз/добу), пінгувач (`CRON_SECRET`).
  - **Заявка з листа (`api/order-from-email.js`):** AI (`extractOrderFromEmail` у `_lib.js`) читає текст листа + вкладення → визначає тип (trade/service/agent), контрагента, суму, позиції, опис → створює `orders` (status `new`, `order_number`=padded count+1); контрагента матчить або **створює** (is_client + contact email); вкладення → `documents` (`source='email'`, OCR до 3 шт) + `order_documents`; `emails.order_id` = нова заявка.
  - **Вихідні (`api/mail-send.js`):** SMTP `nodemailer`, надсилання документа клієнту з вкладенням; кнопка «Надіслати email» у `DocModal` (`components/MailSendModal.jsx`, префіл з `contractor_contacts.email`); пише в `mail_log` + `emails` (direction out, видно в «Пошті»). **Надсилання лишається з розділу Документи** (не з «Пошти»).
  - Авторизація: браузер → Supabase access_token (Bearer, `verifyUser`); cron → `CRON_SECRET`. Серверні операції — service-role (`getAdmin`).
- **Замовлення: Kanban-дошка** (`components/Kanban.jsx`) замість діаграми Ганта. Колонки = статуси: фільтр за типом (trade/service/agent) показує повний pipeline цього типу (з порожніми колонками), змішаний режим — лише наявні статуси в канонічному порядку (`STATUS_ORDER`/`labelForStatus` у `lib/orders.js`). Картка: № + тип + клієнт + сума + наступна дія, підсвітка прострочених. Клік відкриває замовлення (без drag-and-drop — статус рухається кнопкою «Наступна дія»). `Gantt.jsx` видалено.
- **Замовлення: архівування + безпечне видалення** (керування з картки замовлення). Архів (`orders.archived_at`, міграція 008) — відновлюваний, архівні приховані з реєстру й не рахуються в KPI, фільтр-чип «Архів» для перегляду. Видалення — двокрокове підтвердження; **блокується**, якщо до замовлення прив'язані документи (пропонує архів), інакше БД каскадом прибирає КП/субзамовлення, а документи/рухи/листи відв'язуються (FK `SET NULL`).

## 6. Архітектурні рішення (НЕ переробляти)

1. **Additive-еволюція БД на місці** — той самий Supabase-проєкт, без паралельної схеми.
2. **bank_transactions = єдине джерело правди.** Борги = документи − транзакції, авто, ніколи вручну.
3. **Каса = рахунок** (`accounts.type='cash'`), не окремий модуль/таблиця.
4. **Баланс рахує ВСІ неігноровані транзакції** (валідація — лише для P&L, не для «чи рухнулись гроші»). Залишок = `opening_balance` + рухи.
5. **`bank_transactions.direction`** лишається у P&L-семантиці (Доходи/Витрати/ПФД/Інше); in/out обчислюється похідно. (Стара логіка від цього залежала — не міняти на in/out.)
6. **tx↔doc many-to-many через `transaction_documents`** (без таблиці order_transactions). Ланцюг tx → документ → замовлення.
7. **P&L касовий** (закупівлі сидять у `pl_level='cogs'` — без подвійного обліку). FIFO `cost_price` на OUT-рухах доступний для майбутнього accrual.
8. **Переюз робочих lib** зі старої системи (autoClassify, ai, docgen, stockService, vkursi, contractors) — не переписувати з нуля.
9. **Сторінки lazy-load** в App.jsx — важкий `pdfmake` ізольований у чанку Документів.
10. **`dist/` у .gitignore** — Vercel будує з джерела.
11. **Жодних секретів у репо** — `validate.mjs` читає `SUPABASE_SERVICE_KEY` з env. GitHub push-protection увімкнено.
12. **Ролі** (admin/accountant/manager) — усі мають повний доступ; RLS = authenticated повний доступ.
13. **`getAccountBalances()` (lib/accounts.js)** — єдиний розрахунок залишків для Банк/Каса, Дашборду, Бюджету.
15. **Рахунок (invoice) не створює борг** — борг лише з реалізованих документів (накладна/акт), щоб рахунок+накладна не задвоювали. `NON_DEBT_TYPES` у lib/debts.js.
14. **`callClaude()` (lib/ai.js)** — єдиний виклик Claude з авто-повтором; OCR і виписки через нього.
